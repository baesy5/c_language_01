    Chapter 12: 함수로 코드를 리모델링하자 — 업다운 게임 리팩토링¶

    🎯 이 장의 학습 목표¶

    이 장을 마치면 다음을 할 수 있습니다:

    • 기존에 만든 업다운 게임 코드를 여러 함수로 분리하여 재구성할 수 있다
    • 함수 분리의 기준(역할별 분리)을 스스로 판단할 수 있다
    • 함수 선언(프로토타입)의 필요성과 위치를 이해할 수 있다
    • '함수로 나누기 전 vs 나눈 후' 코드를 비교하고 차이를 설명할 수 있다

    예상 학습 시간: 약 2시간

    📚 미리 알아야 할 것¶

    이 장은 이전에 배운 내용을 조합하는 종합 실습 장입니다.

    • Chapter 8: 업다운 게임을 만들었어요 (반복문, 조건문, 랜덤 숫자)
    • Chapter 10~11: 함수의 기본 — 매개변수, 반환값, 함수 만드는 방법

    혹시 Chapter 8의 업다운 게임이 기억나지 않아도 괜찮아요! 이 장에서 원래 코드를 다시 보여드리고, 그걸 함께 고쳐나갈 거예요. 😊

    💡 왜 이걸 배우나요?¶

    여러분, 이런 경험 있으신가요?

    방이 하나뿐인 원룸에 살 때는 옷도, 책도, 식기도 전부 한 공간에 있습니다. 물건이 적을 때는 괜찮지만, 점점 늘어나면 뭐가 어디 있는지 찾기 어렵죠.

    이사를 가서 방이 여러 개인 집으로 옮기면 어떨까요? 침실, 서재, 주방으로 역할별로 물건을 나누면 훨씬 정리가 됩니다.

    프로그래밍도 똑같습니다!

    Chapter 8에서 만든 업다운 게임은 모든 코드가 main 함수 하나에 들어 있었어요. 마치 원룸에 모든 걸 쑤셔넣은 것처럼요. 이제 함수를 배웠으니, 코드를 역할별로 분리해서 깔끔하게 리모델링해 볼 거예요.

    이 과정을 프로그래머들은 리팩토링(Refactoring) 이라고 부릅니다.

    🏠 리팩토링 = 프로그램의 동작은 그대로 유지하면서, 코드의 구조를 더 깔끔하게 정리하는 것

    📚 핵심 개념¶

    개념 1: 리팩토링 — 동작은 그대로, 구조만 바꾸기¶

    🏠 비유로 시작해요

    집 리모델링을 생각해 보세요. 벽을 허물고 방을 나누더라도, 집의 용도(사람이 사는 것)는 변하지 않습니다. 더 쾌적해질 뿐이죠.

    📖 정확한 정의

    리팩토링이란 프로그램의 실행 결과는 전혀 바꾸지 않으면서, 코드를 더 읽기 쉽고 관리하기 쉬운 구조로 개선하는 작업입니다.

    ✏️ 예시로 확인

    우리가 할 작업이 바로 이것입니다: - 리팩토링 전: 업다운 게임을 실행하면 1~100 사이 숫자를 맞추는 게임이 돌아간다 - 리팩토링 후: 똑같은 게임이 돌아간다. 하지만 코드가 훨씬 깔끔하다!

    개념 2: 역할별 함수 분리 — 하나의 함수는 하나의 일만¶

    🍳 비유로 시작해요

    식당을 떠올려 보세요. 좋은 식당은 역할이 나뉘어 있습니다: - 주문 담당: 손님의 주문을 받는다 - 요리 담당: 음식을 만든다 - 서빙 담당: 음식을 가져다준다

    만약 한 사람이 주문도 받고, 요리도 하고, 서빙도 한다면? 느리고 혼란스럽겠죠.

    📖 정확한 정의

    역할별 함수 분리란, 프로그램이 하는 여러 가지 일을 각각 독립된 함수로 나누는 것입니다. 하나의 함수는 하나의 명확한 역할만 담당합니다.

    ✏️ 예시로 확인

    업다운 게임에서 나눌 수 있는 역할:

    graph LR A["🎲 정답 숫자 생성"] --> B["📢 게임 안내 출력"] B --> C["🔍 힌트 제공<br/>UP or DOWN"] C --> D["🎉 결과 출력"]
    • generateAnswer() → 랜덤 정답 숫자를 만든다
    • printWelcome() → 게임 시작 안내 메시지를 출력한다
    • giveHint() → UP인지 DOWN인지 알려준다
    • printResult() → 정답을 맞혔을 때 결과를 출력한다

    개념 3: 함수 프로토타입(선언) — "이런 함수가 있을 거야" 미리 알려주기¶

    📬 비유로 시작해요

    택배를 보내기 전에 송장(운송장)을 먼저 붙이죠? 택배 상자 안에 뭐가 들었는지 모르더라도, 송장만 보면 "누구한테, 어디로 가는 건지" 알 수 있습니다.

    함수 프로토타입은 이 송장 같은 겁니다. 함수의 실제 내용(몸체)이 아래쪽에 있더라도, 위쪽에서 미리 "이런 함수가 존재해!" 라고 알려주는 역할이에요.

    📖 정확한 정의

    함수 프로토타입(함수 선언)이란, 함수의 반환형, 이름, 매개변수 정보만 먼저 컴파일러에게 알려주는 것입니다. 함수 몸체({ } 안의 코드)는 나중에 작성합니다.

    ✏️ 예시로 확인

    // 함수 프로토타입 (선언) — 세미콜론(;)으로 끝남!
int generateAnswer(void);
void giveHint(int answer, int guess);

int main(void) {
    int answer = generateAnswer();  // 아래에 정의된 함수를 여기서 호출 가능!
    // ...
    return 0;
}

// 함수 정의 (실제 내용)
int generateAnswer(void) {
    // 랜덤 숫자를 만들어서 반환
    return rand() % 100 + 1;
}

void giveHint(int answer, int guess) {
    if (guess < answer)
        printf("UP! ⬆️\n");
    else
        printf("DOWN! ⬇️\n");
}

    왜 필요한가요?

    C 언어는 코드를 위에서 아래로 읽습니다. main 함수가 위에 있고 generateAnswer 함수가 아래에 있으면, main에서 호출할 때 컴파일러가 "이게 뭔데?" 하고 에러를 냅니다. 프로토타입을 맨 위에 써주면, "아, 나중에 나올 함수구나!" 하고 미리 알 수 있어요.

    개념 4: Before & After — 왜 함수로 나누면 좋을까?¶

    함수로 나누면 구체적으로 어떤 점이 좋은지 정리해 볼게요.

    🔍 읽기 쉬움: main 함수만 보면 프로그램의 전체 흐름이 한눈에 보입니다

    🔧 수정이 편함: 힌트 메시지를 바꾸고 싶다면? giveHint 함수만 고치면 됩니다

    ♻️ 재사용 가능: generateAnswer 함수를 다른 게임에서도 쓸 수 있습니다

    🐛 버그 찾기 쉬움: "힌트가 이상해!" → giveHint 함수만 집중해서 확인하면 됩니다

    graph TB subgraph BEFORE["❌ 리팩토링 전"] M1["main 함수<br/>━━━━━━━━━━<br/>랜덤 숫자 생성<br/>환영 메시지 출력<br/>반복문으로 입력 받기<br/>UP/DOWN 판단<br/>결과 출력<br/>━━━━━━━━━━<br/>모든 코드가 여기에!"] end subgraph AFTER["✅ 리팩토링 후"] M2["main 함수<br/>━━━━━━━━━━<br/>함수 호출만!<br/>흐름만 보인다"] F1["generateAnswer"] F2["printWelcome"] F3["giveHint"] F4["printResult"] M2 --> F1 M2 --> F2 M2 --> F3 M2 --> F4 end

    💡 잠깐! 여기까지 이해가 잘 되셨나요? 핵심 개념은 4가지뿐이에요. 이제 직접 코드로 만들어 보면서 몸으로 익혀봅시다! 💪

    🔨 따라하기¶

    Step 1: 리팩토링 전 — 원래 코드 확인하기¶

    먼저 Chapter 8에서 만든 업다운 게임 원본 코드를 다시 볼게요. (일부 간소화했습니다)

    #include <stdio.h>
#include <stdlib.h>
#include <time.h>

int main(void) {
    int answer, guess, tries = 0;

    srand(time(NULL));
    answer = rand() % 100 + 1;

    printf("=== 업다운 게임 ===\n");
    printf("1부터 100 사이의 숫자를 맞혀보세요!\n\n");

    while (1) {
        printf("숫자를 입력하세요: ");
        scanf("%d", &guess);
        tries++;

        if (guess == answer) {
            printf("\n🎉 정답! %d번 만에 맞혔습니다!\n", tries);
            break;
        } else if (guess < answer) {
            printf("UP! ⬆️ 더 큰 숫자예요.\n");
        } else {
            printf("DOWN! ⬇️ 더 작은 숫자예요.\n");
        }
    }

    return 0;
}

    이 코드를 잘 보세요. main 함수 안에 모든 것이 들어있습니다: - 랜덤 숫자 생성 - 환영 메시지 출력 - 사용자 입력 + 반복 - UP/DOWN 힌트 - 정답 시 결과 출력

    지금은 30줄 정도라 괜찮아 보이지만, 기능이 늘어나면 순식간에 100줄, 200줄이 됩니다. 그때는 정말 읽기 힘들겠죠?

    Step 2: 역할 나누기 — 어떤 함수로 분리할까?¶

    코드를 읽으면서 "이건 하나의 독립적인 일이다" 라고 느껴지는 덩어리를 찾아보세요.

    분리할 역할 4가지:

    1. 🎲 정답 생성: srand + rand 부분 → generateAnswer() 함수
    2. 📢 환영 메시지: 게임 시작 안내 출력 → printWelcome() 함수
    3. 🔍 힌트 제공: UP/DOWN 판단 + 출력 → giveHint() 함수
    4. 🎉 결과 출력: 정답 맞혔을 때 메시지 → printResult() 함수

    🎯 분리 기준은 간단합니다: "이 코드 덩어리를 한 마디로 설명할 수 있는가?" - "정답 숫자를 만든다" → 한 마디로 설명됨 ✅ → 함수로 분리! - "입력 받고 비교하고 힌트 주고 결과 출력하고..." → 한 마디 안 됨 ❌ → 더 나눠야 함!

    Step 3: 함수 하나씩 만들기¶

    🎲 generateAnswer 함수

    int generateAnswer(void) {
    srand(time(NULL));
    return rand() % 100 + 1;
}
    • 매개변수: 없음 (스스로 랜덤 생성)
    • 반환값: int (1~100 사이 정수)

    📢 printWelcome 함수

    void printWelcome(void) {
    printf("=== 업다운 게임 ===\n");
    printf("1부터 100 사이의 숫자를 맞혀보세요!\n\n");
}
    • 매개변수: 없음
    • 반환값: 없음 (void) — 출력만 하면 됨

    🔍 giveHint 함수

    void giveHint(int answer, int guess) {
    if (guess < answer) {
        printf("UP! ⬆️ 더 큰 숫자예요.\n");
    } else {
        printf("DOWN! ⬇️ 더 작은 숫자예요.\n");
    }
}
    • 매개변수: answer(정답), guess(사용자 추측) — 두 개를 비교해야 하니까!
    • 반환값: 없음 (void) — 출력만 하면 됨

    🎉 printResult 함수

    void printResult(int tries) {
    printf("\n🎉 정답! %d번 만에 맞혔습니다!\n", tries);
}
    • 매개변수: tries(시도 횟수)
    • 반환값: 없음 (void)

    💡 잠깐! 각 함수가 받는 매개변수를 잘 보세요. 함수는 자기 역할에 필요한 정보만 받습니다. printResult는 answer가 필요 없어요. 시도 횟수만 알면 되니까요!

    Step 4: 함수 프로토타입 작성하기¶

    함수들을 main 아래쪽에 정의할 거니까, main 위에 프로토타입을 써줍니다.

    #include <stdio.h>
#include <stdlib.h>
#include <time.h>

// 함수 프로토타입 (선언)
int generateAnswer(void);
void printWelcome(void);
void giveHint(int answer, int guess);
void printResult(int tries);

    이렇게 하면 컴파일러가 "아, 이런 함수들이 나중에 나오는구나" 하고 미리 알 수 있습니다.

    Step 5: main 함수 재구성하기¶

    이제 main 함수를 함수 호출 중심으로 다시 작성합니다.

    int main(void) {
    int answer, guess, tries = 0;

    answer = generateAnswer();
    printWelcome();

    while (1) {
        printf("숫자를 입력하세요: ");
        scanf("%d", &guess);
        tries++;

        if (guess == answer) {
            printResult(tries);
            break;
        } else {
            giveHint(answer, guess);
        }
    }

    return 0;
}

    원래 코드와 비교해 보세요! main 함수만 읽으면:

    1. 정답을 생성한다 (generateAnswer)
    2. 환영 메시지를 출력한다 (printWelcome)
    3. 반복해서 입력받는다
    4. 정답이면 결과 출력 (printResult), 아니면 힌트 제공 (giveHint)

    마치 프로그램의 목차를 읽는 것 같지 않나요? 이것이 함수 분리의 힘입니다! 🎉

    ⚠️ 자주 하는 실수¶

    실수 1: 프로토타입을 안 쓰고 main 아래에 함수를 정의함¶

    // ❌ 에러 발생!
int main(void) {
    int answer = generateAnswer();  // 컴파일러: "이게 뭔데?"
    // ...
}

int generateAnswer(void) {
    // ...
}
    • 왜 이런 일이 생기나요? C 컴파일러는 코드를 위에서 아래로 읽습니다. main에서 generateAnswer를 호출할 때, 컴파일러가 아직 이 함수를 모릅니다.
    • 이렇게 해결해요:
    • 방법 A: main 위에 함수 프로토타입을 작성한다 (우리가 사용한 방법 ✅)
    • 방법 B: 함수 정의를 main 위에 적는다 (간단하지만 코드가 길어지면 main을 찾기 어려워짐)

    실수 2: 프로토타입과 함수 정의의 형태가 다름¶

    // 프로토타입
void giveHint(int answer, int guess);

// 함수 정의 — 매개변수 순서가 다르다!
void giveHint(int guess, int answer) {  // ❌ 순서 바뀜!
    // ...
}
    • 왜 이런 일이 생기나요? 프로토타입을 먼저 쓰고 나중에 함수를 정의할 때, 매개변수 순서나 타입을 다르게 적는 실수가 생깁니다.
    • 이렇게 해결해요: 프로토타입을 복사해서 함수 정의에 붙여넣고, 세미콜론(;)만 { }로 바꾸세요. 그러면 절대 틀리지 않습니다!

    실수 3: 함수 안에서 다른 함수의 변수를 사용하려 함¶

    int main(void) {
    int answer = generateAnswer();
    // ...
}

void giveHint(int answer, int guess) {
    // main의 tries를 쓰고 싶어서...
    printf("%d번째 시도: ", tries);  // ❌ 에러! tries는 main의 변수
}
    • 왜 이런 일이 생기나요? 함수는 각자 독립된 공간을 가집니다. main에서 선언한 tries는 giveHint에서 직접 접근할 수 없어요.
    • 이렇게 해결해요: 필요한 값은 반드시 매개변수로 전달하세요. giveHint에서 tries가 필요하다면 매개변수에 추가합니다.
    void giveHint(int answer, int guess, int tries) {  // ✅ 매개변수로 전달!
    printf("%d번째 시도: ", tries);
    // ...
}
