    📐 게임을 설계하자 — 기획부터 함수 구조 짜기¶

    Part 5 | [2권] 최종 프로젝트 — 내 손으로 게임 완성하기

    🎯 이 장의 학습 목표¶

    이 장을 마치면 다음을 할 수 있습니다:

    • 게임의 전체 흐름을 글과 순서도(Flowchart)로 미리 설계할 수 있다
    • 필요한 함수 목록을 정리하고 각 함수의 역할·입력·출력을 계획할 수 있다
    • 큰 문제를 작은 단위로 나누는 분할 사고(Decomposition)를 실천할 수 있다
    • 설계 문서를 바탕으로 함수 뼈대(스켈레톤) 코드를 작성할 수 있다

    예상 학습 시간: 약 1시간 30분 ⏱️

    📚 미리 알아야 할 것¶

    이전 장까지 배운 내용을 간단히 복습해 볼게요.

    • 함수 정의와 호출: def 함수이름(매개변수): 형태로 함수를 만들고, return으로 값을 돌려주는 방법
    • 조건문과 반복문: if/elif/else, while, for 를 사용해 프로그램 흐름을 제어하는 방법
    • 리스트와 딕셔너리: 여러 데이터를 묶어서 관리하는 방법
    • input() 함수: 사용자로부터 키보드 입력을 받는 방법

    위 개념이 기억나지 않으면, 해당 장을 빠르게 훑어본 뒤 돌아오세요. 이 장은 "코드를 작성하기 전에 생각을 정리하는 방법"을 배우는 시간이에요! 😊

    💡 왜 이걸 배우나요?¶

    레고 블록으로 큰 성을 만든다고 상상해 보세요. 아무 계획 없이 블록을 쌓기 시작하면 어떻게 될까요? 중간에 "아, 여기 문을 넣었어야 했는데…" 하며 다시 허무는 일이 반복될 거예요.

    프로그래밍도 똑같습니다.

    실제 현업 개발자들도 코드를 작성하기 전에 반드시 설계 단계를 거칩니다. 어떤 기능이 필요한지 목록을 만들고, 전체 흐름을 그림으로 그리고, 각 부분을 누가(어떤 함수가) 담당할지 미리 결정하죠.

    이 습관을 지금부터 들이면 여러분은:

    • 🧩 코드 작성이 훨씬 빠르고 수월해집니다
    • 🐛 버그(오류)가 생겼을 때 어디를 고쳐야 하는지 금방 찾습니다
    • 🤝 나중에 다른 사람과 협업할 때도 자연스럽게 소통할 수 있습니다

    이번 장에서는 "가위바위보 게임"을 함께 설계해 볼 거예요. 코드를 한 줄도 쓰지 않고, 오직 생각하고 정리하는 것만으로 게임의 80%를 완성하는 경험을 하게 됩니다!

    📚 핵심 개념¶

    개념 1: 게임 기획서 — 무엇을 만들지 글로 적기 📝¶

    1) 비유로 시작

    요리를 할 때 레시피를 먼저 읽어보는 것처럼, 게임을 만들기 전에 "어떤 게임인지"를 글로 정리하는 것이 게임 기획서입니다. 레시피 없이 요리하면 재료를 빠뜨리거나 순서를 잘못하는 것처럼, 기획서 없이 코딩하면 꼭 빠뜨리는 기능이 생깁니다.

    2) 정확한 정의

    게임 기획서란, 게임의 규칙, 진행 방식, 승리/패배 조건, 사용자가 보게 되는 화면(출력) 등을 글로 정리한 문서입니다. 거창할 필요 없어요. 노트에 몇 줄 적는 것만으로도 충분합니다.

    3) 예시로 확인 — 가위바위보 게임 기획서

    🎮 게임 이름: 가위바위보 대결!

    🎯 게임 목표: 컴퓨터와 가위바위보를 해서 먼저 3번 이기면 승리

    📋 게임 규칙: - 플레이어는 가위, 바위, 보 중 하나를 선택한다 - 컴퓨터도 랜덤으로 하나를 선택한다 - 가위 > 보, 보 > 바위, 바위 > 가위 - 비기면 다시 한다 (무승부) - 먼저 3승을 달성한 쪽이 최종 승리

    🖥️ 화면에 보이는 것: - 게임 시작 환영 메시지 - 매 라운드: 현재 점수, 선택 안내, 결과 - 게임 종료: 최종 결과와 다시 하기 여부

    🔄 게임 종료 후: "다시 하시겠습니까? (y/n)"으로 물어본다

    이렇게 적어두면, 나중에 코딩할 때 "아, 뭘 만들어야 하더라?" 하고 헤매지 않아요.

    개념 2: 순서도(Flowchart) — 흐름을 그림으로 보기 🗺️¶

    1) 비유로 시작

    놀이공원에 가면 입구에 있는 안내 지도를 떠올려 보세요. 지도를 보면 "입구 → 회전목마 → 점심 식사 → 롤러코스터 → 출구" 같은 동선을 한눈에 파악할 수 있죠. 순서도는 프로그램의 안내 지도입니다.

    2) 정확한 정의

    순서도(Flowchart)는 프로그램의 실행 흐름을 도형과 화살표로 표현한 그림입니다.

    • 둥근 사각형: 시작/끝
    • 직사각형: 처리(동작)
    • 마름모: 조건 분기(예/아니오)
    • 화살표: 흐름 방향

    3) 예시로 확인 — 가위바위보 게임 순서도

    flowchart TD A([🎮 게임 시작]) --> B[환영 메시지 출력] B --> C[점수 초기화: 플레이어=0, 컴퓨터=0] C --> D{누군가 3승 했나?} D -- 아니오 --> E[플레이어에게 선택 입력받기] E --> F[컴퓨터 랜덤 선택] F --> G[승패 판정] G --> H[결과와 현재 점수 출력] H --> D D -- 예 --> I[최종 결과 출력] I --> J{다시 하겠습니까?} J -- 예 --> C J -- 아니오 --> K([게임 종료])

    순서도를 그려놓으면 "아, 이 부분에서 반복이 일어나는구나", "여기서 조건 분기가 필요하구나"라는 것이 눈에 보입니다. 코드로 옮기기가 훨씬 쉬워져요!

    💡 잠깐! 순서도를 완벽하게 그릴 필요는 없어요. 핵심 흐름만 잡아도 충분합니다. 나중에 코딩하면서 수정하면 돼요!

    개념 3: 분할 사고(Decomposition) — 큰 문제를 작게 나누기 🧩¶

    1) 비유로 시작

    피자 한 판을 한입에 먹을 수 있을까요? 불가능하죠! 🍕 조각으로 나눠야 먹을 수 있어요. 프로그래밍도 마찬가지입니다. "가위바위보 게임을 만들어라"라는 큰 문제는 한 번에 해결하기 어렵지만, 작은 조각으로 나누면 각각은 아주 쉬워집니다.

    2) 정확한 정의

    분할 사고(Decomposition)란, 복잡한 문제를 해결 가능한 작은 단위로 쪼개는 사고 방식입니다. 컴퓨터 과학에서 가장 중요한 사고법 중 하나이며, 이렇게 나눈 각 조각이 바로 함수가 됩니다.

    3) 예시로 확인

    "가위바위보 게임"이라는 큰 문제를 나눠 볼게요:

    flowchart TD A["🎮 가위바위보 게임<br/>(큰 문제)"] --> B["1. 환영 메시지 보여주기"] A --> C["2. 플레이어 입력 받기"] A --> D["3. 컴퓨터 선택 만들기"] A --> E["4. 승패 판정하기"] A --> F["5. 결과 출력하기"] A --> G["6. 게임 전체 진행하기"] A --> H["7. 다시 할지 물어보기"]

    이렇게 나누면 각 조각은 짧은 함수 하나로 만들 수 있어요. "환영 메시지 보여주기"는 print()만 몇 줄이면 되고, "컴퓨터 선택 만들기"는 random.choice() 한 줄이면 됩니다.

    🍕 기억하세요: "한입 크기"로 나누면 어떤 문제든 해결할 수 있어요!

    개념 4: 함수 설계서 — 각 함수의 역할 미리 정하기 📋¶

    1) 비유로 시작

    회사에서 프로젝트를 할 때, 팀원마다 역할 분담표를 만들죠? "김 대리는 디자인 담당, 이 사원은 자료 조사 담당…" 이렇게요. 함수 설계서도 똑같습니다. 각 함수에게 이름, 맡은 일, 필요한 재료(입력), 결과물(출력)을 미리 정해주는 거예요.

    2) 정확한 정의

    함수 설계서란, 코드를 작성하기 전에 각 함수의 다음 네 가지를 미리 결정한 문서입니다:

    • 함수 이름: 무슨 일을 하는지 알 수 있는 이름
    • 역할: 이 함수가 하는 일 한 줄 설명
    • 매개변수(입력): 함수가 동작하려면 외부에서 받아야 하는 값
    • 반환값(출력): 함수가 일을 끝내고 돌려주는 결과

    3) 예시로 확인 — 가위바위보 게임 함수 설계서

    함수 1: show_welcome() - 역할: 게임 시작 환영 메시지를 화면에 출력한다 - 매개변수: 없음 - 반환값: 없음

    함수 2: get_player_choice() - 역할: 플레이어에게 가위/바위/보 중 하나를 입력받아 돌려준다 - 매개변수: 없음 - 반환값: "가위", "바위", "보" 중 하나 (문자열)

    함수 3: get_computer_choice() - 역할: 컴퓨터의 선택을 랜덤으로 만들어 돌려준다 - 매개변수: 없음 - 반환값: "가위", "바위", "보" 중 하나 (문자열)

    함수 4: decide_winner(player, computer) - 역할: 플레이어와 컴퓨터의 선택을 비교해 승패를 판정한다 - 매개변수: player(플레이어 선택), computer(컴퓨터 선택) - 반환값: "플레이어", "컴퓨터", "무승부" 중 하나 (문자열)

    함수 5: show_round_result(round_num, player, computer, winner, scores) - 역할: 한 라운드의 결과를 보기 좋게 출력한다 - 매개변수: 라운드 번호, 양쪽 선택, 승자, 현재 점수 - 반환값: 없음

    함수 6: play_game() - 역할: 게임 전체를 진행하는 메인 함수 (다른 함수들을 호출) - 매개변수: 없음 - 반환값: 없음

    💡 잠깐! 여기까지 이해가 잘 되셨나요? "아직 코드를 안 썼는데, 벌써 게임이 완성된 것 같은 기분"이 든다면 정상이에요! 이게 바로 설계의 힘입니다. 천천히 다시 읽어봐도 좋아요! 😄

게임 장르: 텍스트 기반 대전 게임

게임 규칙:
1. 플레이어가 가위/바위/보 중 하나를 선택한다
2. 컴퓨터가 랜덤으로 하나를 선택한다
3. 승패를 판정한다 (가위>보, 보>바위, 바위>가위)
4. 비기면 무승부, 점수 변동 없음
5. 먼저 3승한 쪽이 최종 승리

화면에 보여줄 것:
- 시작: "🎮 가위바위보 대결에 오신 것을 환영합니다!"
- 매 라운드: 라운드 번호, 선택 결과, 승패, 현재 점수
- 종료: 최종 승자 발표, 다시 하기 여부

필요한 Python 기능:
- random 모듈 (컴퓨터 선택용)
- input() 함수 (플레이어 입력용)
- while 반복문 (3승까지 반복)
- if/elif/else (승패 판정)

    Step 2: 순서도 그리기 🗺️¶

    목표: 게임의 흐름을 시각적으로 정리합니다.

    종이에 직접 그려도 되고, 앞에서 본 Mermaid 순서도를 참고해도 좋아요. 핵심은 "어떤 순서로 진행되는가"와 "어디서 분기(갈림길)가 생기는가"를 파악하는 것입니다.

    여러분이 그릴 순서도의 핵심 포인트:

    • 반복이 일어나는 곳: "누군가 3승할 때까지" 라운드 반복
    • 조건 분기가 일어나는 곳: 승패 판정, 3승 도달 확인, 다시 하기 여부
    flowchart TD START([시작]) --> WELCOME[환영 메시지 출력] WELCOME --> INIT[점수 초기화] INIT --> CHECK{3승한 사람이<br/>있는가?} CHECK -- 아니오 --> INPUT[플레이어 선택 입력] INPUT --> VALID{유효한<br/>입력인가?} VALID -- 아니오 --> ERRMSG[잘못된 입력 안내] --> INPUT VALID -- 예 --> COMP[컴퓨터 랜덤 선택] COMP --> JUDGE[승패 판정] JUDGE --> SCORE[점수 업데이트] SCORE --> DISPLAY[라운드 결과 출력] DISPLAY --> CHECK CHECK -- 예 --> FINAL[최종 결과 출력] FINAL --> AGAIN{다시<br/>하겠는가?} AGAIN -- 예 --> INIT AGAIN -- 아니오 --> END([종료])

    위 순서도를 자세히 보세요. 코드를 쓰기 전에 이미 프로그램의 모든 흐름이 보이지 않나요?

    특히 주의해서 볼 부분: - "유효한 입력인가?" → 사용자가 엉뚱한 값을 넣을 경우를 대비합니다 - 큰 while 루프: CHECK → INPUT → ... → DISPLAY → CHECK 순환 - 바깥 루프: "다시 하겠는가?"에서 INIT으로 돌아가는 순환

    ✅ 체크포인트: 순서도를 보고 "아, 반복문이 여기 필요하겠다", "여기서 if문을 써야겠다"라는 느낌이 오면 잘 따라오고 계신 거예요!

    Step 3: 분할 사고로 함수 목록 만들기 🧩¶

    목표: 순서도의 각 블록을 함수로 나눕니다.

    순서도를 다시 보면서, 비슷한 역할을 하는 블록을 묶어 봅시다:

    함수로 만들 블록 정리:

    • show_welcome() ← "환영 메시지 출력" 블록
    • get_player_choice() ← "플레이어 선택 입력" + "유효한 입력인가?" + "잘못된 입력 안내" 블록
    • get_computer_choice() ← "컴퓨터 랜덤 선택" 블록
    • decide_winner(player, computer) ← "승패 판정" 블록
    • show_round_result(...) ← "라운드 결과 출력" 블록
    • play_game() ← 나머지 전체 흐름을 관리하는 "감독" 역할
    flowchart LR A["play_game()<br/>🎬 감독"] --> B["show_welcome()<br/>👋 인사"] A --> C["get_player_choice()<br/>🙋 입력받기"] A --> D["get_computer_choice()<br/>🤖 컴퓨터 선택"] A --> E["decide_winner()<br/>⚖️ 판정"] A --> F["show_round_result()<br/>📊 결과 출력"]

    play_game()은 영화 감독처럼, 다른 함수들을 적절한 순서로 호출하는 역할을 합니다. 실제 일은 각 함수가 나눠서 하죠!

    💡 팁: 함수 이름을 지을 때는 "이 함수가 뭘 하는지" 영어로 짧게 적는 것이 좋아요. get_으로 시작하면 "무언가를 가져온다", show_로 시작하면 "화면에 보여준다"는 뜻이에요.

    Step 4: 함수 설계서 완성하기 📋¶

    목표: 각 함수의 입력(매개변수)과 출력(반환값)을 구체적으로 결정합니다.

    아래 내용을 노트에 직접 정리해 보세요:

    show_welcome() - 하는 일: 게임 제목과 규칙 안내 출력 - 입력: 없음 - 출력: 없음 (화면에 출력만)

    get_player_choice() - 하는 일: 가위/바위/보 입력받기, 잘못된 입력이면 다시 물어보기 - 입력: 없음 - 출력: "가위", "바위", "보" 중 하나 반환

    get_computer_choice() - 하는 일: 랜덤으로 가위/바위/보 중 하나 선택 - 입력: 없음 - 출력: "가위", "바위", "보" 중 하나 반환

    decide_winner(player, computer) - 하는 일: 두 선택을 비교해서 누가 이겼는지 판정 - 입력: player(문자열), computer(문자열) - 출력: "플레이어", "컴퓨터", "무승부" 중 하나 반환

    show_round_result(round_num, player, computer, winner, scores) - 하는 일: 이번 라운드 결과를 보기 좋게 출력 - 입력: 라운드 번호(정수), 양쪽 선택(문자열), 승자(문자열), 점수(딕셔너리) - 출력: 없음 (화면에 출력만)

    play_game() - 하는 일: 게임 전체 진행 (환영 → 라운드 반복 → 종료 → 다시하기) - 입력: 없음 - 출력: 없음

    ✅ 체크포인트: 6개 함수의 역할/입력/출력을 모두 정리했나요? 축하합니다! 설계의 80%가 완성되었어요! 🎉
